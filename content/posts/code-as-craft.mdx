## Joints

A cabinetmaker's dovetail joint is stronger than a butt joint, but it also communicates care. Someone understood the forces involved and chose a solution that's both functional and considered.

Code works the same way. The internal structure of a well-written module — naming, flow, boundaries — communicates care to every developer who touches it next. Users never see it. Maintainers live in it.

## Material Knowledge

Every craft requires knowing your materials. A ceramicist knows how clay responds to temperature. A typographer knows how a typeface performs at small sizes.

Software has materials too: languages, frameworks, data structures, protocols. TypeScript behaves differently at the edges of its type system. React's reconciliation has specific performance characteristics. HTTP has semantics that, when respected, make systems predictable.

Good developers treat these as materials to understand, not tools to use. They know where a framework's abstractions leak, where a type system has gaps, where a protocol's guarantees don't hold.

## The Editing Analogy

Film editing is called the invisible art — when it works, you don't notice it. Cuts feel natural, pacing feels inevitable. Behind that are hundreds of decisions about timing, continuity, and rhythm.

Code review is the same. A good refactoring session is pure editing — removing unnecessary complexity, improving flow, making sure the reader's eye moves through the logic without stumbling. The goal: make the complex feel simple.

The best codebases read like well-edited films. Each module does one thing. Transitions between components are clean. Nothing wasted.

## Constraints Produce

Every craft operates within constraints. A sonnet has 14 lines. A film frame has a fixed ratio. A boat hull has to displace its own weight.

In software: performance budgets, type systems, API contracts, browser capabilities. The temptation is to treat constraints as obstacles. The craft approach treats them as design parameters.

A 100ms page load budget isn't a restriction — it's a brief. It forces decisions about what matters, what defers, what gets cut. The constrained solution is almost always better than the unconstrained one would have been.

## Visible Systems

In a well-designed typeface, every character shares a consistent logic — the same stroke weights, curve tensions, optical adjustments. You feel the system without articulating it.

A well-structured codebase has the same quality. Consistent naming, consistent error handling, consistent directory structure. New code almost writes itself because the system guides the decisions.

This is the direct connection between code and craft. Both build systems that are coherent, maintainable, and clear in their intentions. Not beautiful like decoration. Functional like a good tool — balanced, purposeful, obvious in what it does.
